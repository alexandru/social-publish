# I want to change the login model to use sessions (UserSessionsDatabase) instead of current JWT tokens. I want the JWT model purged (when talking about own own endpoints), although HTTP API params will simply become the random token generated by UserSessionsDatabase.

This means that, yes, all protected HTTP endpoints will query the database for seeing if the user is logged in and that's fine.

One thing that I want changed is that, in the context of user authorization, I want internal APIs to use `UserSession` as "context parameter" (see installed skill). This is to force correctness (e.g., some methods/functions should only be used in the context of a logged-in user). One such example is the DocumentsDatabase ... all document queries need to be in the context of a logged-in user, hence all queries need `AND user_uuid = ?`, and to force that, we need to make it clear in the API that a UserSession is required from the "context".

## Current spec

The migration should replace JWT authorization for own endpoints with UUID session tokens backed by UserSessionsDatabase, with no legacy JWT/session compatibility needed. Clients keep using Authorization: Bearer <token>; routes extract the token and call auth.withAuthorization(token) { ... }. UserSessionsDatabase.login owns credential verification via UsersDatabase.verifyPassword, token generation, hashing, and session insertion; persisted sessions store only the SHA-256 token hash and keep the current UserSession schema exactly. UserSessionsDatabase.authorize(rawToken) hashes the provided raw token for lookup and returns UserSession? without retaining the raw token. Sessions expire at a fixed server-side time about 3 months after login using an injectable clock, allow multiple concurrent sessions, support idempotent logout, return 401 for invalid auth cases, and delete expired sessions when encountered. JWT code/config/dependencies should be removed if unused. Internal APIs that need the logged-in user require context(UserSession), and document DB methods derive userUuid only from that context. Tests should cover auth service/routes, session DB behavior, expiry/logout, and document scoping.
